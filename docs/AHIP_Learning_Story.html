The Problem

Before Stage 4, the workflow was correct but explanations were static.

The system already produced deterministic outcomes, but the recommendation layer needed a more readable narrative without changing business logic.

Why it felt limited

Presentation gap
  • Deterministic outputs were reliable, but they were not easy to present in a single clean narrative.
  • The recommendation needed better explanation without allowing the LLM to alter routing, scoring, or audit.
  • The system needed a way to stay auditable while sounding more natural to human reviewers.
The Solution

Only the Consolidator lights up. Everything else stays deterministic.

The LLM joins after safe context is built, then validation and fallback protect the workflow.

Stage 4 adds one controlled LLM enhancement point instead of turning the workflow into a generative system. That keeps the architecture predictable and safe.

Implementation

These are the pieces that made Stage 4 work.

Tap a card to expand the explanation.

It isolates prompts, schemas, provider resolution, service logic, and validation so the Consolidator can call the LLM without leaking responsibilities into the graph nodes.

The prompt contract tells the LLM what to generate, what not to touch, and how to stay inside the structured JSON format required by the validator.

The adapter keeps provider resolution behind one interface, which makes the system easy to swap, test, and protect from circular dependencies.

It behaves like a provider, but returns consistent structured content so the pipeline can be tested without external calls.

Structured validation checks the output format, the response fields, and the protected-field contract before anything is shown to the user.

The service wraps context creation, provider execution, validation, and fallback so the rest of the application only sees a reliable result.

The Consolidator receives deterministic outputs, calls the LLM only for narrative enhancement, and preserves the original recommendation logic.

The response always carries enough metadata to explain what happened, what failed, and whether fallback was used.

Key Features

The system feels simple because the rules stay clear.

A compact summary of the core implementation choices.

Structured OutputEnsures the LLM response is machine-readable before it can influence presentation.
